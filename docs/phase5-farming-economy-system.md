# Phase 5: 种田经营深化系统

## 文档概述

**目标**：在 Demo 0.4.x 基础设施完成后，把现有“可运行的种田闭环”升级为“有经营压力、有内容选择、有短中期目标”的农场循环。  
**版本定位**：Demo 0.5.0  
**前置条件**：Phase 4（EventManager + EffectManager）已完成并通过基础运行验证。  
**Phase 范围**：仅包含 3 个方向

- Phase 5-A：体力与行动消耗系统
- Phase 5-B：多作物与季节经营系统
- Phase 5-C：订单与农场经济反馈系统

本阶段仍然是**种田经营深化**，不是战斗探索阶段，也不是天气/NPC/任务大扩展阶段。

---

## 核心定位

### 1. 当前最缺的是经营压力，不是更多基础设施

Phase 4 已经把跨系统通信、效果、四季、存档等底座接上。  
继续堆基础设施的收益已经开始下降。

现在真正缺的是：

- 玩家每天为什么不能无脑重复操作
- 玩家为什么要在不同作物之间做选择
- 玩家为什么要为了更高收益做计划

所以 Phase 5 的重点不是“更强架构”，而是“更强经营感”。

### 2. 先深化农场，再切探索

完整独立游戏最终会包含四成战斗探索，但当前不宜立刻切到战斗最小闭环。  
如果农场这条主循环还没有基本经营压力，战斗探索上线后只会放大整体失衡。

Phase 5 的正确顺序是：

1. 先加入体力限制
2. 再扩作物选择和季节经营
3. 再用订单系统给玩家短中期目标

### 3. Phase 5 仍然坚持数据驱动

本阶段新增内容必须优先接入现有配置体系：

- `ConfigManager`
- `CropConfig`
- `ShopConfig`
- `GameManager.export_save_data()`
- `SaveManager` 编排式存档

不允许重新回到“把内容写死在单个脚本里”的旧路径。

---

## Phase 5-A：体力与行动消耗系统

### 目标

让现有农场日常第一次具备明确的行动成本，使“今天先做什么、做到什么程度、是否睡觉结束一天”成为决策。

### 核心交付

- `GameManager` 新增体力状态与 API
- 农场动作按类型消耗体力
- 体力不足时动作失败并给出明确反馈
- 次日开始或睡觉后恢复体力
- UI 显示当前体力
- 体力进入存档/读档链路

### 本阶段不做

- 饥饿、口渴、疲劳多条资源
- 行走消耗体力
- 战斗生命值系统
- 食物恢复体系

### 工程原则

- 体力只服务当前农场玩法
- `FarmInteractionSystem` 负责动作前检查，不把体力逻辑散落到各地块
- `SaveManager` 仍只做编排，不新增单独的 stamina 存档模块

---

## Phase 5-B：多作物与季节经营系统

### 目标

把当前单一小麦链路扩成一个真正有选择的轻量经营系统。

### 核心交付

- 扩展到 3 至 5 种作物
- 每种作物有明确定位差异
- `CropConfig` 承担更多经营字段
- 商店购买、背包显示、播种与收获链路适配多作物
- 季节适配真正影响播种决策

### 本阶段不做

- 大规模内容库
- 复杂肥料系统
- 天气系统
- 高级品质/稀有度体系

### 工程原则

- 继续使用 `crop_config_id`
- 去掉当前地块和 UI 中对 `seed_wheat` / `crop_wheat` 的硬编码依赖
- 新作物全部通过 `resources/data/crops/` 接入

---

## Phase 5-C：订单与农场经济反馈系统

### 目标

让农场经济从“卖掉所有作物”升级为“在即时售卖和计划性目标之间做选择”。

### 核心交付

- 新增轻量订单系统
- 每日或节气刷新一组订单
- 订单奖励高于直接售卖，但需要玩家预留产出
- UI 中可查看、提交、结算订单
- 订单状态进入存档/读档链路

### 本阶段不做

- 完整 NPC 委托网络
- 多阶段剧情任务
- 稀有订单池和复杂权重系统
- 联动天气、探索掉落、建筑升级的大型经济网

### 工程原则

- 订单先围绕现有农产品展开
- `OrderManager` 作为轻量系统即可，不需要过早抽象成任务框架
- 订单系统必须复用现有 `GameManager`、`SaveManager`、`UIRoot` 风格

---

## 与当前系统的正确关系

### 1. GameManager

Phase 5 的核心状态中心仍然是 `GameManager`。  
新增体力、订单奖励结算、更多物品流转时，都要优先围绕 `GameManager` 现有接口演进，而不是旁路建立一套新状态树。

### 2. FarmInteractionSystem

Phase 5-A 中它成为动作前置验证的第一入口：

- 检查工具是否可执行
- 检查地块是否可执行
- 检查体力是否足够
- 成功后再扣体力并执行动作

### 3. Plot

Phase 5-B 的重点不是让 `Plot` 更复杂，而是让它从当前的单作物硬编码中解耦。  
地块仍然负责：

- 当前种了什么
- 当前状态是什么
- 生长如何推进

但不应该继续写死“小麦就是唯一作物”。

### 4. ShopUI / InventoryUI / UIRoot

Phase 5 不是只改数据层。  
经营选择必须在 UI 上对玩家可见：

- 体力要可见
- 多作物要可见
- 订单目标要可见

否则玩法升级会停留在内部逻辑层。

---

## 验证目标

### Prompt 14 完成后

- 玩家不能无限执行农活
- 睡觉和次日开始的恢复逻辑正确
- 存档读档后体力不丢失
- 玩家能清楚知道动作失败是因为体力不足

### Prompt 15 完成后

- 农场里至少存在 3 种有效作物选择
- 商店、背包、播种、收获、售卖链路全部跑通
- 季节限制对经营产生真实影响

### Prompt 16 完成后

- 玩家除了直接售卖外，还有订单目标可追
- 订单不会破坏现有经济平衡
- 订单状态存档后可恢复

---

## 结论

Phase 5 的正确路线是：

- 先补经营压力
- 再补经营选择
- 再补经营目标

这样下一阶段如果切入战斗探索，农场主循环才已经具备“可独立成立”的基础。
