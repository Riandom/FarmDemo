## 本次更新概览

本次更新把项目从“纯农场经营 Demo”推进到了“**经营准备 -> 进入洞窟 -> 飞镖战斗 -> 带回掉落**”的最小闭环。

这次实际接入的重点有 8 个：

1. `Main` 现在承担农场和洞窟两种区域状态切换，见 `scripts/app/main.gd:186`、`scripts/app/main.gd:197`
2. 农场新增洞窟入口与战备商人
3. 玩家可在洞窟中使用飞镖进行俯视即时战斗，见 `scripts/app/main.gd:225`
4. 洞窟为单房间原型，含撤离点、史莱姆、蝙蝠和基础掉落
5. 战斗掉落直接进入现有背包，不另做地面拾取系统
6. 战备商店复用了现有商店 UI，但通过 `ui_type` 区分为独立模态
7. 洞窟掉落已接入普通商店售卖链路
8. 存档版本提升为 `0.7.0`，并开始保存当前世界区域状态，见 `scripts/systems/save_manager.gd:8`

当前原型的目标不是“完整战斗系统”，而是先验证这条资源流是否成立：

> 种田卖货 -> 买飞镖 -> 下洞战斗 -> 带回掉落 -> 卖掉或留作后续成长

## 玩家现在能体验到的内容

从玩家视角看，当前已经接入的内容是：

- 在农场内靠近洞窟入口，按 `F` 进入洞窟
- 在农场内靠近战备商人，按 `F` 打开飞镖补给商店
- 购买 `基础飞镖` 后，飞镖会作为热栏物品参与当前装备体系
- 进入洞窟后，手持飞镖并按 `F`，会向朝向方向发射投射物
- 洞窟里当前有两种敌人：
  - 史莱姆：慢速追击，适合练习命中和站位
  - 蝙蝠：更快，更像近距离扑击压力
- 敌人死亡后会直接把掉落发进背包
- 洞窟内有撤离点，可主动返回农场
- 玩家生命降到 0 时会被送回农场，并承担轻惩罚
- 带回来的洞窟掉落可以在普通商店售卖换钱

当前体验上的特色不是“近战挥剑”，而是：

- 飞镖是显式消耗资源
- 每次下洞前都要先做补给准备
- 战斗压力主要来自弹药消耗和出征成本，而不是高额死亡惩罚

## 核心闭环说明

当前最小闭环已经成立，但仍然很轻量。

### 农场准备阶段

- 玩家通过作物和订单获取金币
- 玩家找战备商人购买 `dart_basic`
- 飞镖进入现有背包 / 热栏体系，不单独开战斗背包

### 洞窟探索阶段

- 玩家通过入口进入单房间洞窟
- 玩家在洞窟里用飞镖进行远程战斗
- 史莱姆和蝙蝠会追击并造成接触伤害
- 击杀敌人后，掉落直接进入背包

### 返回与结算阶段

- 主动撤离：安全返回农场，保留当前收益
- 战败返回：回到农场，扣少量金币和飞镖
- 带回的 `矿石碎片`、`洞窟精粹` 已可卖给普通商店

这条闭环的价值在于，它已经证明：

- 农场经济可以驱动战斗准备
- 战斗收益可以回流进现有经济系统
- 热栏、背包、商店、存档不需要另起一套战斗专用系统

## 系统实现拆解

### 场景流转：`Main` 负责双区域控制

当前没有采用独立切场景的方式，而是让 `Main` 同时管理农场区和洞窟区。

关键实现点：

- `scripts/app/main.gd:186` `enter_cave()`
- `scripts/app/main.gd:197` `exit_cave()`
- `scripts/app/main.gd:241` `_set_world_area()`

当前策略是：

- 农场对象仍存在于主场景
- 洞窟对象挂在 `ExploreRoot`
- 切换时通过显示/隐藏与区域状态切换完成

这样做的原因是：

- 现有系统已经以主场景直连方式构建
- 可以最大限度复用玩家、UI、背包和交互逻辑
- 当前原型重点是闭环验证，不是复杂场景管理

代价是：

- 农场和洞窟目前仍共享同一个运行根场景
- 后续如果要做多层洞窟，可能需要再进一步拆分

### 战斗行为：飞镖通过现有输入桥转入战斗动作

当前没有新增一套专门的“攻击键”。  
仍然沿用现有 `interact` 流程：

- 如果前方有交互目标，就执行交互
- 如果前方没有交互目标，就尝试执行战斗动作

战斗动作入口：

- `scripts/actors/player/player_input_bridge.gd`
- `scripts/app/main.gd:225` `player_use_combat_item()`

当前规则：

- 只有在 `current_world_area == "cave"` 时才允许飞镖攻击
- 当前装备必须是 `dart_basic`
- 每次发射消耗 1 发飞镖
- 投射物由 `ProjectileDart` 负责飞行和命中

### 洞窟运行时：`CaveController` 负责房间生命周期

核心控制器：

- `scripts/world/explore/cave_controller.gd:22` `set_cave_active()`
- `scripts/world/explore/cave_controller.gd:58` `spawn_dart()`
- `scripts/world/explore/cave_controller.gd:77` `_on_enemy_defeated()`

它当前负责：

- 洞窟房间激活与隐藏
- 洞窟中敌人的启停和复位
- 飞镖投射物的容器管理
- 击杀敌人后的掉落发放

当前房间是：

- 单房间
- 固定布局
- 固定敌人布点

这是一种明确的原型策略：先验证交互和资源流，不提前做随机地图或多层结构。

### 敌人与掉落：先做最小差异，不做复杂 AI

当前敌人：

- `scripts/world/explore/enemy_slime.gd`
- `scripts/world/explore/enemy_bat.gd`

行为差异：

- 史莱姆：慢速接近，基础压力
- 蝙蝠：更快，更适合测试投射物距离控制

掉落策略：

- 史莱姆偏向 `ore_fragment`
- 蝙蝠偏向 `cave_essence`
- 掉落直接调用 `GameManager.add_item()`

当前不做地面拾取的原因：

- 这是额外的一套对象生命周期和交互链
- 对验证“战斗收益是否能回流现有经济系统”不是必需
- 先直接入包，能更快暴露背包、商店、存档是否兼容

### 商人与 UI：战备商人复用 `ShopUI`

当前战备商店不是另写一套购买界面，而是基于现有商店系统复用。

关键点：

- `scenes/ui/combat_vendor_ui.tscn`
- `scripts/ui/shop_ui.gd`
- `ui_type = "combat_vendor"`

之所以必须加 `ui_type`，是因为：

- 如果仍然硬编码成 `"shop"`，UIRoot 的模态状态会混乱
- 战备商店虽然复用同一套组件，但逻辑上必须是独立模态

这次复用的好处：

- 筛选、商品显示、金币扣除、背包加物品都直接沿用
- 当前只需要单独配置卖什么，不需要重做交易 UI

### 存档与状态：开始保存世界区域

这次 Phase 6 原型新增了世界区域状态：

- `scripts/systems/game_manager.gd:11` `signal world_area_changed`
- `scripts/systems/game_manager.gd:260` `set_current_world_area()`

存档版本：

- `scripts/systems/save_manager.gd:8` `SAVE_VERSION = "0.7.0"`

当前存档扩展的重点不是保存复杂洞窟进度，而是先保存：

- 玩家当前在哪个区域
- 读档后主场景应该恢复成农场还是洞窟状态

这样做可以避免：

- 存档读档后 UI 和区域显示不一致
- 玩家仍在洞窟坐标，但场景逻辑已经回到农场

## 已知限制与风险

这部分很重要。当前原型已经接入，但还不能假设“一切都稳定”。

### 当前限制

- 还没有经过完整 Godot 编辑器运行验证
- 洞窟仍是单房间原型，不含多层结构
- 飞镖的“低耐久”当前主要表现为持续补给压力，不是独立耐久条
- 掉落直接进背包，未做地面拾取
- 战备商店复用了 `ShopUI`，后续可能需要独立视觉和交互布局

### 当前风险点

- 主场景双区域切换是否会和现有农场交互残留产生冲突
- 飞镖是否总能正确进入热栏友好的槽位
- 读档恢复区域状态时，是否会出现显示与逻辑不同步
- 战败返回是否会和已有 UI 模态、时间推进或存档链路冲突

## 手动测试清单

建议按下面顺序测试，不要跳测。

1. 农场入口与战备商人交互
   - 入口是否显示正确提示
   - 战备商人是否能打开独立商店面板
2. 飞镖购买后是否进入热栏
   - 购买后是否能通过 `Q/E` 或 `1~0` 选中
3. 洞窟中飞镖是否可发射并消耗
   - 在洞窟中按 `F` 是否发射
   - 数量是否同步减少
4. 敌人是否追击与伤害
   - 史莱姆和蝙蝠是否都有明显行为差异
5. 掉落是否进入背包
   - 击杀后是否立即增加 `ore_fragment` / `cave_essence`
6. 撤离与战败返回
   - 主动撤离是否回到农场
   - 战败是否轻惩罚回农场
7. 普通商店是否可售卖洞窟掉落
   - 回农场后是否能卖出掉落换钱
8. 存档读档后区域状态是否正确
   - 在农场和洞窟分别读档，都要验证一次

## 下一步建议

如果当前原型跑通，下一步不要立刻堆更多敌人或地图，优先做下面三类收口：

1. **稳定性收口**
   - 修编辑器实际运行中暴露的问题
   - 把存档 / UI / 区域切换跑稳

2. **体验收口**
   - 调整飞镖价格、掉落价值、战败惩罚
   - 让“出征准备”更清晰

3. **系统深化**
   - 增加第二种战斗补给或特殊飞镖
   - 让洞窟掉落开始参与更明确的成长解锁
   - 再考虑是否拆独立洞窟场景管理

当前最重要的结论是：

> Prompt 18 已经把 Phase 6 的第一条战斗探索闭环接进来了，但它仍然是“原型”，不是“稳定战斗系统”。
