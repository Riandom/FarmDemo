# Phase 7：世界区域、镇子与 NPC 基础层

## 文档目的

本阶段文档用于定义 `Phase 7` 的目标、边界和推荐实现顺序。  
它不是具体实现 Prompt，也不是对外介绍文案，而是后续生成 `Prompt 19` 到 `Prompt 23` 的母版。

当前项目已经完成：

- `Phase 5`：种田经营深化
- `Phase 6`：飞镖出征式战斗与单房间洞窟探索原型

当前项目还已经完成 `Phase 7` 的第一轮骨架接入：

- `Prompt 19`：储物箱、材料分类与农场生活支撑
- `Prompt 20`：`farm / house / town / cave` 统一区域切换
- `Prompt 21`：小屋与镇子基础布局及稳定入口
- `Prompt 22`：NPC 交互、基础对话与简单日程
- `Prompt 23`：好感度、送礼、NPC 委托与镇子商店承载

接下来不继续扩大战斗玩法，而是转向“世界结构 + 镇子/NPC 基础层”。

## 当前实现状态

本文件最初用于定义 `Phase 7` 的计划顺序。  
截至当前版本，`Prompt 19 -> 23` 已全部完成第一轮接入，因此本文件现在同时承担“阶段总纲 + 已实现边界索引”的作用。

当前可以确认已落地的范围：

1. 农场、小屋、镇子、洞窟已纳入同一套区域结构
2. 小屋已承担床和主储物箱等生活锚点
3. 镇子已具备 3 个基础 NPC 和基础 hub 布局
4. NPC 已能提供对话、简单日程、送礼与关系值变化
5. 现有订单和商店系统已经开始由 NPC 承载一部分入口与人格化来源

当前尚未扩展的部分：

- 复杂 NPC 剧情树
- 复杂寻路和完整日程模拟
- 镇子内多建筑深度功能
- 更丰富的 NPC 委托分支
- 战斗系统的继续特色化

## 为什么现在暂停战斗扩展

当前战斗原型已经具备最小闭环：

- 农场赚钱
- 购买飞镖
- 进入洞窟
- 即时战斗
- 带回掉落
- 战败轻惩罚返回

但战斗特色仍未定型。此时继续扩写敌人、地图或武器，只会增加错误方向的沉没成本。

相比之下，镇子与 NPC 更适合作为下一阶段主轴，原因是：

- 能显著提升生活感和世界感
- 能承接现有商店、订单、金币、背包和时间系统
- 能为后续剧情、关系、委托和功能建筑提供稳定承载层
- 不会阻断未来重新定义战斗特色

因此本阶段的策略是：

- 战斗系统冻结在原型阶段
- 仅做 bugfix 和兼容维护
- 正式进入世界区域、镇子、小屋和 NPC 基础层

## Phase 7 的核心目标

本阶段要完成的不是“大量新玩法”，而是把游戏世界从“单点玩法原型”升级成“可承载生活模拟的空间结构”。

核心目标有 5 个：

1. 建立统一的世界区域结构
2. 把农场、小屋、镇子、洞窟纳入同一套区域切换骨架
3. 为镇子地图和 NPC 提供稳定的空间承载
4. 让 NPC 成为现有商店、订单和经营反馈的具象入口
5. 保证未来新增内容不需要重写区域与交互底层

## 世界区域结构

本阶段先固定 4 个区域：

- `farm`
- `house`
- `town`
- `cave`

其中：

- `farm`：主经营场景，承担种植、出货、农场内摆放等核心日常循环
- `house`：农场内室内区域，承担睡觉、私人储物和后续生活化功能
- `town`：生活 hub，承担 NPC、商店、委托和日常交流
- `cave`：探索区域，暂时保留现有战斗原型

### 关于 `house`

产品上，小屋属于农场。  
技术上，小屋应视为独立区域。

这样处理的原因：

- 室内外出生点和显示规则不同
- 后续家具、床、私人储物箱、室内互动会更清晰
- 不会把农场主场景塞得越来越重

## 世界区域与场景切换系统

`Phase 7` 的底层骨架是：

**世界区域与场景切换系统**

它负责的不是地图美术本身，而是：

- 当前所在区域的统一状态
- 区域进入和退出
- 入口与出生点定位
- 区域切换后的 UI 显示差异
- 存档与读档后的区域恢复
- 未来新增区域时的统一接入方式

### 技术原则

- 先采用轻量方案，不立即上复杂异步加载框架
- `GameManager.current_world_area` 继续保留
- 由专门的区域管理层统一驱动区域切换
- 小屋、镇子、洞窟都走同一套进入接口
- 洞窟先并入统一区域系统，但不在本阶段新增战斗玩法

### 本阶段不做

- 多区域异步流式加载
- 大地图连续无缝移动
- 多层洞窟拓展
- 区域内复杂事件调度器

## 镇子与 NPC 的定位

镇子不是“另一张地图”，而是世界结构中承接生活感和经营反馈的核心 hub。

第一版镇子的定位：

- 小型 hub
- 可稳定进入和返回
- 有基础商店和日常 NPC
- 支持对话、服务和后续委托

NPC 在本阶段的作用：

- 提供生活存在感
- 承接已有系统入口
- 为后续关系值、送礼和委托打底

第一版 NPC 不追求复杂剧情树，而是先做到：

- 能被识别
- 能交互
- 有基础对话
- 有简单按时段站位
- 能承载商店或服务功能

## 与现有系统的关系

本阶段必须尽量复用现有底层，而不是重写。

### 需要直接承接的现有系统

- `GameManager`
- `SaveManager`
- `TimeManager`
- `EventManager`
- 背包/热栏系统
- 普通商店与战备商店 UI 体系
- 订单系统

### 本阶段对这些系统的要求

- 不推翻现有金币与背包链路
- 不重写商店底层交易逻辑
- 不废弃现有订单系统
- 用 NPC 和镇子场景承载现有能力，而不是另起一套系统

换句话说，后续应该是：

- “商店由 NPC 提供”
- “部分订单来自 NPC”
- “镇子成为功能入口”

而不是：

- 再做一套新的商店
- 再做一套新的委托
- 再做一套新的区域状态

## 本阶段推荐开发顺序

### Prompt 19：农场生活支撑系统补完

目标：

- 补齐储物箱
- 稳定物品分类，加入 `材料`
- 为后续小屋、礼物、委托和镇子功能做仓储与整理准备

重点：

- 只补硬缺口
- 不扩写大量新农业玩法

### Prompt 20：世界区域与场景切换系统

目标：

- 统一 `farm / house / town / cave`
- 统一进入、离开、出生点和区域恢复逻辑
- 把现有洞窟纳入同一套区域结构

重点：

- 先做骨架
- 不在这一步塞镇子剧情或 NPC 关系逻辑

### Prompt 21：镇子地图、小屋接入与基础入口布局

目标：

- 接入农场小屋
- 接入镇子地图
- 配置农场、小屋、镇子、洞窟之间的稳定入口和返回点

重点：

- 先建立可往返空间
- 不急着堆叠大量内容点

### Prompt 22：NPC 交互框架、基础对话与简单日程

目标：

- 建立统一 NPC 交互方式
- 接入基础对话 UI
- 做简单按时间段站位或状态切换

重点：

- 第一版不做复杂分支剧情
- 第一版不做复杂寻路

### Prompt 23：好感度、送礼、NPC 委托与镇子商店承载

目标：

- 建立基础关系值
- 接入送礼
- 让部分订单人格化为 NPC 委托
- 用镇子 NPC 承接现有商店能力

重点：

- 尽量复用现有订单和商店底层
- 不重写经济链路

## 明确不在本阶段做的内容

为了防止阶段目标被冲散，本阶段明确不做：

- 战斗系统特色扩写
- 新武器体系
- 新敌人大量追加
- 多层洞窟或大规模探索地图
- 动物养殖
- 烹饪
- 加工机体系
- 复杂天气系统
- 复杂剧情分支树
- 复杂 NPC 寻路和完整日程模拟

这些内容后续都可能做，但不属于 `Phase 7` 的首要目标。

## 阶段完成标准

当以下条件成立时，可以认为 `Phase 7` 骨架完成：

- 玩家能稳定在农场、小屋、镇子、洞窟之间切换
- 小屋作为独立区域存在且能承接生活功能
- 镇子能承载基础 NPC 和功能入口
- NPC 能承担对话、商店或简单服务
- 好感度、送礼和 NPC 委托已与现有经营系统接通
- 现有种田、订单、背包、商店和洞窟原型没有因区域系统接入而回归损坏

## 审查要点

在生成具体 Prompt 之前，应先确认以下决策没有歧义：

- 是否接受 `house` 技术上为独立区域
- 是否接受战斗系统在本阶段冻结
- 是否接受 `Prompt 19 -> 23` 的顺序固定
- 是否接受镇子第一版仅为小型 hub
- 是否接受 NPC 第一版不做复杂剧情和复杂寻路

如果这些点都成立，就可以按本文件顺序继续生成具体 Prompt。

## 当前阶段结论

`Phase 7` 的第一轮基础骨架已经成立。  
后续开发不再需要回头重做 `Prompt 19 -> 23`，而应在此基础上继续做：

- 文档与索引维护
- 回归验证
- 镇子/NPC 层细化
- 后续新阶段规划
