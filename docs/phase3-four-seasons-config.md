# Phase 3: 完整四季循环与数据配置系统

## 概述

**目标**：在 Demo 0.2 的春季时间系统基础上，完成 Demo 0.3.0 的四季框架升级。  
本阶段分两步实施：

1. `Prompt 10`：数据配置系统
2. `Prompt 11`：完整四季循环

本版的产品边界固定如下：

- 以**四季框架优先**为目标
- 不引入完整新作物玩法链
- 季节视觉只做**轻量方案**
- 配置系统采用**增量接管**

---

## 总体设计原则

### 1. 增量演进，不推翻已稳定链路

Demo 0.2 的主链路已经稳定，因此 Phase 3 不做一次性底层翻修。  
实现原则：

- 优先统一配置访问入口
- 保留旧资源和旧回退逻辑
- 在现有可运行链路之上逐步替换业务来源

### 2. 配置驱动优先

季节和时间参数都要资源化，不再继续在 `TimeManager` 中堆业务常量。  
统一通过 `ConfigManager` 读取：

- 作物配置
- 工具配置
- 季节配置
- 时间系统配置

### 3. 可扩展性优先于一次性“做完”

本版只完成后续系统的底座：

- 四季循环
- 季节倍率
- 适种季节判断
- 存档兼容
- 轻量季节 UI

暂不纳入：

- 粒子系统
- 天气系统
- 事件系统
- 完整新作物玩法链

---

## Phase 3-A：Prompt 10 数据配置系统

### 目标

建立统一配置入口，为四季系统提供基础设施。

### 核心交付

- 新增 `ConfigManager` autoload
- 新增 `SeasonConfig`
- 新增 `TimeSystemConfig`
- 新增四季 `.tres` 配置
- 新增时间系统 `.tres` 配置
- `CropConfig` 增加 `suitable_seasons`

### 关键约束

- `ConfigManager` 采用增量接管，不做一次性重构
- `time_config.tres` 在 0.3.0 内保留，作为兼容回退
- `TimeManager` / `FarmInteractionSystem` / `Plot` 都优先通过 `ConfigManager` 取配置
- 若配置缺失，系统必须给 warning 而不是直接崩溃

### 不纳入本阶段

- 完整新作物玩法链
- 删除旧时间配置
- 联机配置同步

---

## Phase 3-B：Prompt 11 完整四季循环

### 目标

把当前“只有春季”的时间系统扩展为完整四季循环系统。

### 核心交付

- `TimeManager` 支持春夏秋冬完整推进
- 新增 `season_changed(season_id, year_count)` 信号
- 新增 `year_count`
- 季节倍率通过 `SeasonConfig` 驱动作物生长
- `Plot` 生长接口升级为支持增长单位
- `CropConfig.suitable_seasons` 进入播种限制
- 时间 UI 与季节弹窗更新
- 季节色调渐变
- 0.2.0 旧档兼容

### 四季倍率

| 季节 | 倍率 | 规则 |
|------|------|------|
| 春季 | 1.0 | 基础速度 |
| 夏季 | 1.2 | 生长更快 |
| 秋季 | 0.9 | 生长稍慢 |
| 冬季 | 0.0 | 室外停止自然生长 |

### 关键约束

- `ConfigManager` 只负责配置访问，不承担业务执行
- 季节视觉只做轻量方案：色调 + 弹窗
- 不引入粒子系统
- 不引入完整新作物玩法链

---

## 技术边界与兼容策略

### 时间系统

- 时间总规则来自 `TimeSystemConfig`
- 季节参数来自 `SeasonConfig`
- `TimeManager` 保留旧 `time_config.tres` 的兼容回退，直到新链路稳定

### 生长系统

- 废弃“只按天推进一次阶段”的刚性模型
- 改为基于增长单位/倍率推进 `growth_progress`
- 后续天气、肥料、温室都在这条接口上扩展

### 存档系统

- `SaveManager` 外部 API 不变
- `TimeManager.export_save_data()` 增加 `year_count`
- 读取 0.2.0 存档时缺失该字段默认回退到 `0`

---

## 验证目标

### Prompt 10 完成后

- `ConfigManager` 启动成功
- 所有现有 crop/tool 资源可通过统一入口访问
- Demo 0.2 主链路不退化

### Prompt 11 完成后

- 春夏秋冬完整闭环
- 冬后回春时年份累加
- 夏季快于春季，秋季慢于春季，冬季停滞
- 播种季节限制正确生效
- 时间 UI 和季节弹窗正确显示
- 0.2.0 旧档兼容读取正常

---

## 结论

Phase 3 的正确路线是：

- 先统一配置访问入口
- 再做四季循环
- 保留兼容回退
- 控制视觉范围

这条路线更稳，也更适合作为未来天气、事件、更多作物和 NPC 系统的底座。
