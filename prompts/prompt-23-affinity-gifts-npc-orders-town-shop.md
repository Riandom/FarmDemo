# Phase 7-E：好感度、送礼、NPC 委托与镇子商店承载实现提示词

## 任务概述

在 NPC 基础交互稳定后，本阶段正式把镇子与 NPC 接回现有经营循环。

目标是把以下系统接起来：

- `好感度`
- `送礼`
- `NPC 委托`
- `镇子商店承载`

这一步完成后，现有的：

- 种田
- 背包
- 商店
- 订单
- 时间系统

将不再只是抽象功能，而会通过 NPC 和镇子空间被具象化。

---

## 核心定位

### 1. 先做轻量关系系统，不做剧情树

第一版关系系统的目标只是：

- 让 NPC 有可累积的关系值
- 让日常交流和送礼有反馈
- 为未来剧情和事件预留空间

不做：

- 恋爱系统
- 复杂剧情锁
- 分支事件链
- 节庆事件网

### 2. 送礼是关系系统的第一条显性行为

如果只有好感度数值，没有可执行动作，玩家感知会很弱。  
因此第一版必须允许：

- 玩家持有物品与 NPC 交互送礼

并得到至少三档反馈：

- 喜欢
- 普通
- 不喜欢

### 3. NPC 委托应复用现有订单系统

当前项目已经有订单系统。  
本阶段不应推翻它重做一套新系统，而应让它“人格化”。

即：

- 现有部分订单由某个 NPC 发布
- UI 上体现来源 NPC
- 提交后同时影响奖励与关系值

### 4. 商店功能应由 NPC 承载，而不是继续做抽象入口

现有普通商店和战备商店底层已经存在。  
本阶段重点不是重写交易系统，而是：

- 让它们由镇子 NPC 提供入口和身份

这样世界感才成立。

---

## 交付范围

### 必做内容

- 每个基础 NPC 拥有好感度数值
- 日常对话可提供微量好感度提升
- 玩家可向 NPC 送礼
- 送礼反馈分为至少 3 档
- 现有订单系统支持“订单发布者 NPC”
- 部分订单改为 NPC 委托
- 镇子内至少 1 个商店 NPC 正式承载普通商店功能

### 本阶段不做

- 恋爱与婚姻
- 好感事件分支剧情
- 多档礼物动画表现
- 节日系统
- 镇子建筑升级系统
- 复杂委托链

```text
scripts/world/npc/
├── npc_relationship_runtime.gd

└── npc_order_bridge.gd

scripts/systems/
└── order_manager.gd

scripts/ui/
├── dialogue_ui.gd
└── order_board_ui.gd
```

说明：

- 不要求严格按这些文件拆，但“关系 / 礼物 / 委托桥接”职责必须清晰
- `OrderManager` 要尽量增量扩展，而不是推翻重写

---

## 好感度系统要求

### 数据结构

每个 NPC 至少拥有：

```gdscript
npc_id: String
affinity: int
```

第一版不要求复杂等级表，但至少要有清晰的上限和下限约束。

### 增长来源

第一版建议只接入两种增长来源：

- 每日首次对话
- 成功送礼

### 行为要求

- 同一天首次对话可以增加少量好感
- 同一天重复对话不重复加分
- 好感变化应有文本反馈或 UI 提示

---

## 送礼系统要求

### 基本规则

玩家持有物品并与 NPC 交互时，应允许触发送礼行为。

第一版建议：

- 通过对话 UI 中的“赠送当前手持物品”入口
- 或在交互中检测当前热栏物品并给出赠送选项

重点是：

- 逻辑清晰
- 不引入过重 UI

### 喜好分级

每个 NPC 至少支持：

- `like`
- `neutral`
- `dislike`

可按物品 ID 做轻量映射，不要求现在就做外部复杂配置表。

### 结果要求

送礼成功后至少会：

- 扣除 1 个对应物品
- 返回文本反馈
- 修改好感度

若物品不允许赠送或没有持有量，应明确提示。

---

## NPC 委托要求

### 与订单系统的关系

`OrderManager` 仍然是委托底层真相来源。  
本阶段的改造重点是：

- 在订单结构中新增 `publisher_npc_id`
- UI 显示“委托来自谁”
- 部分订单由 NPC 发布

### 第一版委托规则

建议：

- 保留部分匿名订单
- 同时加入部分 NPC 委托

这样过渡更稳，不会一口气推翻现有订单体验。

### 提交收益

NPC 委托提交成功后：

- 继续给金币
- 可额外给少量好感度

先不要扩出复杂多奖励链路。

---

## 镇子商店承载要求

### 核心目标

当前已有普通商店和战备商店底层逻辑。  
本阶段要做的是：

- 镇子中的商店 NPC 打开这些现有 UI

也就是说，变化应主要发生在入口承载层，而不是交易底层。

### 第一版最小要求

至少有一个镇子 NPC 能承载：

- 普通商店

若后续战备商店暂时仍保留原入口，也允许，但应开始向“NPC 承载”方向靠拢。

---

## 存档与兼容要求

### 需要保存的内容

至少应保存：

- 每个 NPC 的好感度
- 每日对话是否已加分
- 订单的发布者信息

### 兼容策略

旧档没有 NPC 关系数据时：

- 应安全回退到默认值

旧订单没有 `publisher_npc_id` 时：

- 应视为匿名订单

---

## 验收标准

- 与 NPC 每日首次对话会产生关系值变化
- 玩家可成功送礼并获得明确反馈
- 不同礼物能触发喜欢 / 普通 / 不喜欢差异
- 部分订单显示来源 NPC
- 成功完成 NPC 委托后可获得金币和关系收益
- 镇子商店由 NPC 入口承载且交易正常
- 不破坏现有背包、订单、商店、时间与存档链路

---

## 实现建议

### 建议优先顺序

1. 先补 NPC 关系值存储
2. 再接每日首次对话加分
3. 再接送礼
4. 再把订单系统扩成支持 `publisher_npc_id`
5. 最后把普通商店正式挂到镇子 NPC

### 推荐做法

- 关系系统先小而稳
- 礼物偏好先做轻量映射
- 委托尽量复用现有订单 UI 和逻辑

### 不推荐做法

- 为了“角色感”重写整套订单系统
- 一开始就做很复杂的好感分层剧情
- 一开始就让所有 NPC 都承担复杂功能
