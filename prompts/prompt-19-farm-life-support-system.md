# Phase 7-A：农场生活支撑系统补完实现提示词

## 任务概述

在正式进入世界区域、镇子和 NPC 之前，先为 FarmDemo 补齐一层“生活支撑系统”。

本阶段的目标不是继续扩写农业玩法，也不是提前做镇子内容，而是先解决这几个会直接阻碍后续开发的问题：

- 背包物品量继续增长后缺少稳定仓储
- `ore_fragment`、`cave_essence` 这类物品还没有被正式纳入“材料类”
- 后续小屋、送礼、NPC 委托和镇子商店都需要更稳定的物品分类与交付基础

因此本阶段的核心内容固定为：

- `储物箱系统`
- `材料分类接入`
- `农场生活支撑补完`

这是 Demo `0.8.x` 进入 `Phase 7` 之前的承上启下阶段。

---

## 核心定位

### 1. 储物箱不是附属装饰，而是后续生活层的必要基础

当前项目已经有：

- 50 格背包
- 多作物
- 洞窟掉落
- 订单
- 战斗补给

如果没有仓储，后续一旦接入：

- 小屋
- 礼物
- 材料积累
- 镇子商店
- NPC 委托

玩家的背包会很快失控。

所以本阶段的储物箱必须作为正式系统接入，而不是临时测试对象。

### 2. “材料”必须成为正式分类

当前 `ConfigManager.get_item_category()` 已经有：

- `tool`
- `seed`
- `crop`
- `other`

但随着洞窟掉落和后续镇子功能进入，`other` 会很快失去意义。  
因此本阶段必须把：

- `material`

提升为正式分类。

第一批应归入 `material` 的内容至少包括：

- `ore_fragment`
- `cave_essence`

后续新增镇子、委托、制作或礼物物品时，也应优先走这一类，而不是继续塞进 `other`。

### 3. 本阶段只补底层支撑，不引入新大系统

本阶段不是：

- 小屋系统
- 区域切换系统
- 镇子 NPC 系统
- 加工机系统
- 动物养成

本阶段只做“让这些东西以后能稳接进来”的基础准备。

## 交付范围

### 必做内容

- 农场内 1 个可交互储物箱
- 储物箱独立存储槽位数据
- 储物箱 UI，可在背包和箱子间存取物品
- 储物箱存档与读档
- `material` 分类正式接入
- 背包筛选加入 `材料`
- 商店筛选加入 `材料`
- 当前掉落物和后续材料物品走统一分类规则

### 本阶段不做

- 多个可放置箱子的建造系统
- 箱子搬动/拆除
- 家具摆放
- 小屋内部场景
- 区域切换
- 礼物系统
- NPC 委托系统
- 制作台/锻造台
- 自动整理背包按钮

## 储物箱系统要求

### 第一版产品形态

第一版只放 1 个固定储物箱在农场中，位置可先手工摆放在主场景。  
目的不是做箱子摆放玩法，而是先验证：

- 存储
- 读取
- 交互
- UI
- 存档

### 储物箱数据结构

储物箱必须拥有独立槽位容器，不与玩家背包混用。

建议第一版采用：

- `10 × 5`
- 共 `50` 格

这样能直接复用现有背包的槽位心智模型，也方便后续小屋私人储物统一体验。

每个槽位结构应与玩家背包一致：

```gdscript
{}
```

或：

```gdscript
{
    "item_id": "ore_fragment",
    "count": 4
}
```

### 储物箱唯一标识

第一版虽然只有 1 个箱子，但仍必须支持唯一标识，例如：

```gdscript
@export var chest_id: String = "farm_chest_01"
```

原因：

- 后续小屋箱子、镇子仓库、多人箱子都依赖这层标识
- 现在不加，后面会重构存档结构

### 必须提供的接口

储物箱脚本至少应提供：

```gdscript
func get_slots() -> Array
func set_slots(new_slots: Array) -> void
func export_save_data() -> Dictionary
func apply_save_data(data: Dictionary) -> void
```

不要求现在就做复杂容器抽象，但要保证未来可以扩到多个箱子。

## 储物箱 UI 要求

### UI 目标

玩家打开储物箱后，必须能直接完成：

- 从背包放入箱子
- 从箱子取回背包

第一版不追求拖拽和华丽视觉，只要求操作稳定、信息清楚。

### 推荐布局

建议采用双栏布局：

- 左侧：玩家背包
- 右侧：储物箱

每侧都使用固定格子展示。

为避免界面过大：

- 储物箱 UI 可复用现有槽位逻辑
- 但单格尺寸应比常规背包界面更紧凑

### 交互规则

第一版建议采用最稳妥的规则：

- 点击玩家背包槽位：将整组物品转移到箱子中的首个可用位置
- 点击储物箱槽位：将整组物品转移回背包中的首个可用位置
- 不做拆分堆叠
- 不做拖拽
- 不做右键半组移动

### UI 接入要求

储物箱 UI 应接入 `UIRoot` 的模态体系，至少支持：

- `open_ui()`
- `close_ui()`
- `ui_opened`
- `ui_closed`

并保持与：

- 背包
- 商店
- 订单
- 暂停菜单

互斥正常。

## 材料分类接入要求

### 正式分类集合

本阶段后，分类应统一为：

- `tool`
- `seed`
- `crop`
- `material`
- `other`

### ConfigManager 要求

`ConfigManager.get_item_category(item_id)` 必须成为唯一可信入口。  
不要在多个 UI 文件里自己再写一套判断。

分类规则建议为：

- `ToolConfig` -> `tool`
- `CropConfig.seed_item_id` -> `seed`
- `CropConfig.harvest_item_id` -> `crop`
- `ItemConfig.category == "material"` -> `material`
- 其余 -> `other`

### 第一批材料物品

至少把以下物品配置正式归为 `material`：

- `ore_fragment`
- `cave_essence`

如果 `dart_basic` 当前是普通消耗品，则不强行归入 `material`，仍可保留为 `other`。

## 背包与商店 UI 调整要求

### InventoryUI

当前背包筛选按钮必须从：

- `全部 / 工具 / 种子 / 作物`

扩展为：

- `全部 / 工具 / 种子 / 作物 / 材料`

要求：

- 只增加筛选能力
- 不改变现有槽位背包结构
- 不破坏热栏逻辑

### ShopUI

商店筛选也应同步支持 `材料`。  
原因是后续镇子商店和洞窟掉落售卖都需要这一层。

要求：

- 普通商店可识别材料类物品
- 售卖区显示材料时分类正确
- 现有种子/作物/工具交易逻辑不回归

---

## 存档与兼容要求

### 1. 储物箱存档

存档中应新增储物箱数据块，至少能保存：

- `chest_id`
- `slots`

即使第一版只有 1 个箱子，也不要把它写成孤立字段，例如：

```gdscript
"farm_chest_slots": [...]
```

更稳妥的结构是数组或映射，例如：

```gdscript
"storage_chests": [
    {
        "chest_id": "farm_chest_01",
        "slots": [...]
    }
]
```

### 2. 旧存档兼容

旧档没有储物箱数据时：

- 不应报错
- 默认生成空箱子

### 3. 版本策略

如果当前版本为 `0.7.0`，本阶段可升到：

- `0.8.0`

但必须继续兼容读取 `0.7.x`。

## 与后续阶段的关系

本阶段完成后，后续阶段可以直接复用这些成果：

- `Prompt 20`
  - 区域切换时，小屋和农场箱子都能走同一存档链路
- `Prompt 21`
  - 小屋可直接接入已有储物箱系统
- `Prompt 22`
  - NPC 对话与交互不需要再处理一套杂乱物品分类
- `Prompt 23`
  - 委托、礼物、材料收集和镇子商店能直接识别 `material`

所以本阶段必须克制，但要把基础打稳。

## 验收标准

### 储物箱

- 农场中存在 1 个可交互储物箱
- 靠近后有明确提示
- 按 `F` 可打开储物箱 UI
- 玩家可把物品从背包放入箱子
- 玩家可把物品从箱子取回背包

### UI

- 储物箱 UI 与现有模态系统兼容
- 打开储物箱时不会和背包/商店/订单 UI 打架
- 关闭储物箱后输入状态恢复正常

### 分类

- 背包筛选中能看到 `材料`
- 商店筛选中能看到 `材料`
- `ore_fragment`、`cave_essence` 被正确识别为 `material`

### 存档

- 存档后退出再读档，箱子物品保持正确
- 旧档无箱子数据时可正常读取
- `0.7.x` 存档不会因为本阶段变更失效

### 回归

- 热栏、背包、商店、订单系统仍正常
- 洞窟掉落进入背包与售卖链路不受破坏
- 农场地块、体力、时间推进、睡觉读档不回归

## 实现建议

### 建议优先顺序

1. 先补 `material` 分类和物品配置
2. 再实现储物箱节点与独立槽位数据
3. 再接储物箱 UI
4. 最后接入存档与回归验证

### 推荐做法

- 尽量复用现有 50 格槽位逻辑
- 尽量复用现有 UIRoot 模态管理方式
- 尽量复用 `GameManager` 的物品增减接口

### 不推荐做法

- 再做一套和背包完全不同的箱子物品结构
- 把箱子数据写死在 `GameManager` 的玩家背包字段里
- 在多个 UI 内各自实现一套分类判断
- 为了一个箱子就提前引入复杂容器继承体系
